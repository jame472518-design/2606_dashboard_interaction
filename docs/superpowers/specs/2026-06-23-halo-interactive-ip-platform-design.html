Halo 互動 IP 平台 · 設計文件

日期：2026-06-23 狀態：設計已確認，待寫實作計畫 硬體：AMD Strix Halo · 32GB · Windows 11 場域：展覽 / 體驗館 / 觀光

1 · 目標與範圍

在店內擺一台 Halo，作為可互動的 IP 角色。資料存本地，顧客可與 IP 對話。平台需多客戶可抽換：不同客戶 = 不同資料夾，統一首頁切換；設定外置，方便換模型與加客戶。

核心需求

  • 全本地：離線可用、無月費、資料不外流
  • 多租戶：加客戶／改人設／換模型只動資料夾與設定檔
  • 分階段：先文字 kiosk，再疊加語音、形象、手機端
  • 雙語：繁體中文 + 英文
  • IP 形象一致：忠實承載客戶官方外型，不 AI 即時生臉

非目標（YAGNI）

  • 不做雲端後台 / 帳號系統
  • 不落地顧客個資（僅短期對話記憶）
  • Phase 1 不做語音、不做 Live2D

2 · 整體架構

店內螢幕 (Kiosk 瀏覽器, 全螢幕) │ ▼ 本地 Web App ──► Orchestrator (對話協調層) │ ┌───────────────┼────────────────┐ ▼ ▼ ▼ 本地 LLM 知識庫 (RAG) 人設 / 形象規則 (Ollama) 向量檢索+結構化資料 (persona + ip_rules)
設計原則：平台程式碼、客戶資料、設定三者解耦。LLM／知識庫／人設互不影響；語音與形象是後續「插件式」疊加，不動核心。

技術選型（Phase 1）

元件選擇理由
LLM 執行Ollama (Windows)安裝簡單、API 穩定、Strix Halo 可用 Vulkan/ROCm 加速
對話模型Qwen2.5 14B（起步）32GB 跑得順、中英雙語強
知識庫結構化 Markdown + RAG回答有依據、不亂編
Embeddingbge-m3全本地、中英文皆強
前端本地網頁全螢幕 kiosk後續階段沿用
後端單一本地服務（實作計畫定案）協調 LLM + RAG + 人設

3 · 部署模式（單一 server + 模式切換）

同一套程式碼，用 config/global.yaml 的 mode 決定開機行為：

hub 首頁列出所有 tenants，自由切換（開發／展示用）
single 鎖定單一 tenant，開機直接進入該 IP，不顯示其他客戶（擺到客戶店裡時用）

# config/global.yaml
mode: hub                 # hub 首頁可切換 / single 鎖定一個客戶
active_tenant: museum-A   # single 模式時生效
llm:
  provider: ollama
  model: qwen2.5:14b      # ← 換模型只改這裡
  base_url: http://localhost:11434
embedding:
  model: bge-m3
server: { port: 8080, default_lang: zh-TW }
不採用「每客戶獨立 process/port」：實際每台 Halo 在店裡只服務一個客戶，single 模式即可達成隔離。

4 · 目錄結構

halo-ip-platform/
├── config/
│   └── global.yaml          # 全域：mode、預設 model、embedding、port、語言
├── tenants/                  # 每個客戶/產品一個資料夾
│   ├── _template/            # 新客戶範本（複製即用）
│   ├── museum-A/
│   │   ├── tenant.yaml       # 此客戶：名稱、語言、model 覆寫、ui
│   │   ├── character/        # IP 形象包（見 §6）
│   │   ├── persona/          # 人設 / system prompt
│   │   ├── knowledge/        # 結構化知識資料 (md)
│   │   ├── index/            # 向量索引（build 產物，可重建）
│   │   └── assets/           # logo 等
│   └── expo-B/ ...
├── server/                   # 後端協調服務
└── web/
    ├── home/                 # 統一首頁（hub 模式列出 tenants）
    └── chat/                 # 對話頁
# tenants/museum-A/tenant.yaml
display_name: "海洋博物館"
persona_file: persona/guide.md
languages: [zh-TW, en]
llm: { model: qwen2.5:32b }   # 覆寫全域；沒寫就吃 global 預設
ui:
  default_view: showcase      # showcase 全螢幕形象 / chat 直接開對話
  voice_first: true           # 預設語音優先（Phase 2）
  chat_panel: { position: bottom, collapsible: true }
  portrait:  { chat_scale: corner }

5 · 人設與知識庫資料結構

原則：人設一個檔、知識一條一個檔，純文字 Markdown，非工程人員可編輯增刪。

人設檔 persona/guide.md

---
name_zh: 小海
name_en: Coral
voice: 親切、口語、略帶俏皮
languages: [zh-TW, en]
---
## 說話風格
- 一次講重點。
- 不確定就說「不太確定」，
  引導去服務台 — 絕不亂編。
## 開場白
- zh-TW: 「嗨～我是小海！」
- en: "Hi! I'm Coral."

知識庫 knowledge/ — 一條一檔

---
id: exhibit-002
type: exhibit   # exhibit/faq/info/story
title: 深海發光生物區
lang: zh-TW
zone: 2F-A
tags: [深海, 生物發光, 水母]
---
這一區展示深海中會發光的生物……
  • 英文版另存 exhibit-002.en.md，依提問語言檢索
  • type: info（時間/票價/廁所）→ 首頁快捷鈕
索引 build：掃 knowledge/ → 切塊 → 算 embedding → 存本地向量庫。資料增修後重跑一次，不碰程式碼。

6 · IP 形象包（維持外型一致）

每個 tenant 一個 character/，平台忠實承載客戶官方 IP，不 AI 即時生成角色臉。

character/
├── character.yaml   # 形象規格(單一真相來源)
├── reference/       # 官方設定圖/表情表(角色聖經)
├── portrait/        # P1 顯示用立繪
├── expressions/     # idle/talking/happy/thinking
└── live2d/          # P3 依官方外型製作
# character.yaml
brand:
  primary_color: "#0A6E8C"
  accent_color:  "#FFD23F"
  logo: character/reference/logo.png
visual:
  expressions:
    idle:    .../idle.png
    talking: .../talking.png
ip_rules:                # 注入系統 prompt
  - 自稱固定用「小海」，不可改名
  - 不做違反角色設定的發言

維持外型一致的三原則

  • 不 AI 生臉：用官方立繪 + 表情圖依對話狀態切換 → 外型 100% 忠於原作
  • 品牌套全畫面：配色/字體/logo 套進整個 kiosk UI
  • 跨階段不走樣：Live2D 以官方設定圖為本去 rig；ip_rules 鎖名字與底線

客戶圖片輸入分級

層級客戶提供Phase 1Phase 3
Tier 1 最低1 張高解析立繪（透明 PNG）直接用單張圖做會說話頭像（SadTalker 類）／生成表情經核可
Tier 2 較好立繪 + 一組表情圖表情切換、零生成表情更自然
Tier 3 最佳PSD 分層原檔—正規 Live2D rig（眨眼/轉頭/嘴型同步）
一張圖即可起步；Live2D 關鍵在「分層」，素材清單把 PSD 分層檔列為加分項。圖片是設計素材，非餵給 LLM 的 runtime 輸入。自帶 IP 與沒有 IP 的客戶都支援。

7 · Phase 1 MVP — 畫面與資料流

可收縮雙狀態畫面

狀態 A — 形象全螢幕（預設、沉浸）

┌───────────────────────────┐ │ [logo] [中/EN] │ │ ┌─────────┐ │ │ │ IP │ │ │ │ 立繪 │ ← 佔滿 │ │ │ │ 會動表情│ │ └─────────┘ │ │ 「點我打字，或對我說話」 │ │ ◦ 對我說話 ⌨ 打字 │ └───────────────────────────┘

狀態 B — 展開對話（點打字後滑出）

┌───────────────────────────┐ │ [logo] [中/EN] │ │ ┌──┐ 對話內容(一問一答) │ │ │IP│ ────────────────── │ │ └──┘ ────────────────── │ │ ┌───────────────────────┐ │ │ │ 對話框 / 輸出區 │ │ │ └───────────────────────┘ │ │ [時間][票價][動線] │ │ ┌────────────┐[送出][▼收合]│ │ └────────────┘ │ └───────────────────────────┘

兩狀態行為由 tenant.yaml 的 ui 區塊客製（預設視圖、面板位置、是否可收合、形象縮放、語音優先）。

對話資料流

顧客輸入(中/EN)
  → Orchestrator：
      1. 判定語言（或用切換鈕）
      2. query → embedding(bge-m3)
      3. 從該 tenant 的 index 檢索 top-k 知識條目（對應語言）
      4. 組 prompt：人設 + ip_rules + 檢索到的知識 + 近幾輪對話
  → Ollama (Qwen2.5) 回答
      ├─ 有命中知識 → 依據回答，立繪切「講話」
      └─ 沒命中    → fallback：「這我不太確定，可以問服務台喔」
  → 串流顯示回答 → 回到待機表情

關鍵規則

  • 強制 grounding：檢索不到相關知識走 fallback，不讓模型瞎掰
  • 短期記憶、不存個資：只留當前顧客近幾輪；閒置逾時（約 60 秒）清空回待機
  • 快捷鈕來自知識庫：type: info 條目做成一鍵按鈕
  • 回應速度：本地 14B 約數秒；串流逐字顯示讓等待有感
Phase 1 驗收：一台 Halo 全螢幕開機自動進入 → 選定 tenant（hub 切換 / single 鎖定）→ 顧客用中/英文字問展館問題、得到有依據回答、立繪切表情 → 換客戶/改人設/更新知識/換模型全靠改資料夾與設定檔。

8 · 分階段路線圖

Phase 1
多租戶文字 kiosk + RAG + 形象切換 + 設定外置 — 自己寫（市面無現成多租戶 RAG kiosk）
Phase 2
語音：本地 ASR (Whisper 類) + TTS — 整合 Open-LLM-VTuber 元件（AMD 友善、MIT、吃 Ollama）
Phase 3
會動會講話的虛擬形象 — Live2D（有 PSD 分層）或單張圖會說話頭像（SadTalker 類）
Phase 4
QR → 手機端，Halo 當店內 WiFi 本地伺服器、多人同時用 — 沿用後端 API

借 vs 自己做（網路調研結論）

項目決策說明
多租戶 + 設定外置 + RAG自己做市面無此組合，是核心差異
語音 + Live2D（P2/P3）借 Open-LLM-VTuberMIT、支援非 NVIDIA GPU/CPU，適配 Strix Halo
RAG 博物館導覽參考VirtualMuseumGuide
Handcrafted Persona Engine不用雖為 kiosk/博物館用途，但只支援 Windows + NVIDIA CUDA，Strix Halo（AMD）用不了
